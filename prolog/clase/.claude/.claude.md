## 🚨🚨🚨 ERROR #1 DEL USUARIO: NO INSTANCIAR ANTES DE USAR 🚨🚨🚨

**ESTE ERROR LE PASA CONSTANTEMENTE. Siempre alertar cuando vea este patrón.**

El usuario SIEMPRE tiende a poner predicados que **USAN** variables **ANTES** de predicados que las **INSTANCIAN**.

**Patrón problemático universal:**
```prolog
% ❌ MALO: Usa X antes de instanciarla
algo(...) :- USA(X), INSTANCIA(X).

% ✓ BUENO: Instancia X antes de usarla
algo(...) :- INSTANCIA(X), USA(X).
```

**Ejemplos concretos del usuario:**
```prolog
% ❌ append antes de recursión
inorder(bin(I,R,D), L) :- append(LI, [R|LD], L), inorder(I,LI), inorder(D,LD).
%                         ↑ USA LI y LD           ↑ INSTANCIA LI y LD (tarde)

% ✓ recursión antes de append
inorder(bin(I,R,D), L) :- inorder(I,LI), inorder(D,LD), append(LI, [R|LD], L).
%                         ↑ INSTANCIA LI y LD           ↑ USA LI y LD

% ❌ comparación antes de append
algo(...) :- length(H,1), H \= [X], append(H,L,XS).
%                         ↑ USA H    ↑ INSTANCIA H (tarde)

% ✓ append antes de comparación
algo(...) :- length(H,1), append(H,L,XS), H \= [X].
%                         ↑ INSTANCIA H    ↑ USA H
```

**REGLA DE ORO:** Preguntarse siempre: "¿Esta variable ya tiene valor cuando la uso?"

## ⚠️ REGLA CRÍTICA: Orden de Metas

**Variables deben estar INSTANCIADAS antes de usarse en predicados generadores.**

Predicados generadores (append, length con lista libre, etc.) generan infinitas soluciones con variables no instanciadas. Esto causa bucles infinitos.

**Orden correcto:**
1. **RESTRINGIR** (length, guardas, condiciones)
2. **INSTANCIAR** (recursión, cálculos)
3. **USAR** (predicados generadores con variables ya instanciadas)

**Ejemplo:**
```prolog
% ❌ MALO: append(L2,H,YS) antes de reverse(L1,L2) → L2 libre → ∞
reverse(XS,YS) :- append(H,L1,XS), append(L2,H,YS), reverse(L1,L2).

% ✓ BUENO: reverse(L1,L2) instancia L2 antes de append(L2,H,YS)
reverse(XS,YS) :- append(H,L1,XS), reverse(L1,L2), append(L2,H,YS).
```

**IMPORTANTE:** Cuando el usuario pregunte "por qué esto se traba/buclea/no termina", SIEMPRE explica con:
1. **Árbol de resolución SLD completo** mostrando dónde ocurre el bucle
2. **Traza detallada** de la ejecución paso a paso
3. **Por qué la primera solución funciona** (si aplica) y luego falla en backtracking
4. **Comparación con la versión correcta**

## ⚠️ ERROR FRECUENTE DEL USUARIO: Comparar Variables No Instanciadas

El usuario tiende a poner comparaciones (`\=`, `=`, etc.) **ANTES** de instanciar las variables con `append` u otros predicados.

**Patrón problemático recurrente:**
```prolog
% ❌ MALO: compara ANTES de instanciar
length(H,1), H \= [X], append(H,L,XS), ...
             ↑
         H = [_A] (libre) → \= falla porque [_A] SÍ puede unificar con [X]

% ✓ BUENO: instancia ANTES de comparar
length(H,1), append(H,L,XS), H \= [X], ...
                              ↑
                          H = [1] (instanciada) → [1] \= [2] funciona
```

**Regla para el usuario:**
> "Primero INSTANCIA con append/length/etc, DESPUÉS COMPARA con \= o ="

**Nota:** Este error le ha pasado múltiples veces. Cuando vea código con comparaciones antes de append, alertar inmediatamente.

## ⚠️ ERROR FRECUENTE: `member` con listas duplicadas

`member(X, Lista)` genera **un choice point por cada aparición** de X en la lista.

**Ejemplo:**
```prolog
?- member(1, [1,1,1]).
true ;
true ;
true.    % 3 soluciones (3 veces aparece 1)
```

**Problema:** Si usás `member` en un predicado recursivo sobre una lista con duplicados, obtenés múltiples soluciones idénticas.

**Solución:** Usar `list_to_set/2` para eliminar duplicados antes de `member`:
```prolog
list_to_set(L2, L2SinDuplicados), member(H, L2SinDuplicados)
```

## ⚠️ ERROR FRECUENTE: Olvidar cubrir AMBOS casos

Si escribís una regla con una condición, **solo cubre cuando la condición es verdadera**.

**Ejemplo problemático:**
```prolog
% Solo cubre cuando H ESTÁ en L2
intersección([H|T], L2, [H|R]) :- member(H, L2), ...
% ¿Y qué pasa cuando H NO está en L2? ❌ NO HAY REGLA
```

**Solución:** Agregar regla para el caso contrario con `not`:
```prolog
intersección([H|T], L2, [H|R]) :- member(H, L2), ...       % H está
intersección([H|T], L2, R) :- not(member(H, L2)), ...      % H NO está
```

**Moraleja:** Siempre preguntarse: "¿Qué pasa si esta condición NO se cumple?"

## ⚠️ ERROR FRECUENTE: Typos en nombres de variables

```prolog
intersección([H1|T1], L2, L3) :- ... member(H, L2) ...
              ↑                              ↑
             H1                              H  ← ¡Variable diferente!
```

**Problema:** Prolog NO detecta esto como error. H queda como variable libre.

**Moraleja:** Revisar que los nombres de variables sean consistentes entre la cabeza y el cuerpo de la regla.

## ⚠️ ERROR FRECUENTE: `append` con lista resultado LIBRE

Cuando usás `append(LA, [H|LB], L2)` y **L2 está libre**, genera infinitas opciones:

**Ejemplo problemático:**
```prolog
permutación([H|T], L2) :-
    append(LA, [H|LB], L2),    % ❌ L2 libre → infinitas opciones
    append(LA, LB, L3),
    permutación(T, L3).

?- permutación([1,2], P).
P = [1,2] ;        % Primera solución ✓
(bucle infinito)   % Intenta LA=[_], LA=[_,_], LA=[_,_,_], ...
```

**Por qué pasa:** `append(LA, [H|LB], L2)` con L2 libre genera:
- LA=[], L2=[H|LB]
- LA=[_A], L2=[_A,H|LB]
- LA=[_A,_B], L2=[_A,_B,H|LB]
- ... ∞

**Solución:** Restringir la longitud de L2 antes de usar append:
```prolog
permutación([H|T], L2) :-
    length([H|T], N),           % L1 tiene N elementos
    length(L2, N),              % ✓ L2 también (esto instancia L2)
    append(LA, [H|LB], L2),     % Ahora L2 está instanciada
    append(LA, LB, L3),
    permutación(T, L3).
```

**Regla:** Si vas a "buscar un elemento en una lista" con append y la lista está libre, **primero restringí su longitud con `length`**.

## ⚠️ ERROR FRECUENTE: RECURSIÓN ANTES DE DIVISIÓN

**PATRÓN CRÍTICO:** Cuando construís una estructura desde una entrada, primero DIVIDÍ la entrada, después RECURSÁ.

**Ejemplo problemático (arbolConInorder):**
```prolog
% ❌ MALO: Recursión antes de división
arbolConInorder(Lista, bin(I, R, D)) :-
    arbolConInorder(LI, I),        % ❌ LI LIBRE → infinitas opciones
    inorder(I, LI),                % Tarde para instanciar
    arbolConInorder(LD, D),        % ❌ LD LIBRE → más infinitas opciones
    inorder(D, LD),
    append(LI, [R|LD], Lista).     % Debería ir PRIMERO

% ✓ BUENO: División antes de recursión
arbolConInorder([], nil).
arbolConInorder(Lista, bin(I, R, D)) :-
    append(LI, [R|LD], Lista),    % ✓ PRIMERO: divide Lista (finito)
    arbolConInorder(LI, I),       % ✓ DESPUÉS: LI instanciada
    arbolConInorder(LD, D).       % ✓ DESPUÉS: LD instanciada
```

**Por qué la versión mala genera recursión infinita:**

1. `arbolConInorder(LI, I)` con LI **libre** puede:
   - LI = [] → I = nil (opción 1)
   - LI = libre → I = bin(...) → más recursión (opción 2)
   - Opción 2 genera más opciones infinitas recursivamente

2. No hay límite sobre qué tan grande puede ser LI porque **nunca se restringe**.

3. Al hacer backtracking después de la primera solución, intenta construir árboles cada vez más profundos infinitamente.

**Patrón abstracto:**
```prolog
% ❌ MALO: Recursión antes de división
predicado(Input, Resultado) :-
    predicado(ParteA, SubResultadoA),  % ParteA libre → ∞ opciones
    predicado(ParteB, SubResultadoB),  % ParteB libre → ∞ opciones
    dividir(ParteA, ParteB, Input).    % ¡Demasiado tarde!

% ✓ BUENO: División antes de recursión
predicado(Input, Resultado) :-
    dividir(ParteA, ParteB, Input),    % Input instanciado → partes finitas
    predicado(ParteA, SubResultadoA),  % ParteA instanciada → finito
    predicado(ParteB, SubResultadoB).  % ParteB instanciada → finito
```

**Moraleja:** El append es tu amigo para dividir, pero debe ir **ANTES** de las llamadas recursivas, no después. Si construís algo desde una entrada conocida, **primero desarmá la entrada** en partes finitas.

## Errores Comunes y Patrones Problemáticos

### 1. Predicados Generadores con Variables No Instanciadas

**Problema crítico:** Usar predicados generadores (que producen infinitas soluciones) con variables no instanciadas ANTES de instanciarlas causa bucles infinitos.

**Predicados generadores problemáticos:**
- `append(X, Y, Z)` - con 2+ argumentos no instanciados
- `length(L, N)` - con L no instanciada
- Predicados recursivos que generan infinitas soluciones

**Ejemplo: reverse/2 con orden incorrecto**

```prolog
% ❌ ORDEN MALO - NO TERMINA
reverse(XS, YS) :-
    append(H,L1,XS),      % H puede ser cualquier cosa
    append(L2,H,YS),      % ❌ L2 NO INSTANCIADA → ∞ backtracking
    length(H,1),          % Nunca se alcanza
    reverse(L1, L2).      % Nunca se alcanza

% ✓ ORDEN CORRECTO - TERMINA
reverse([], []).
reverse(XS, YS) :-
    length(H,1),          % 1. Restringe H primero
    append(H,L1,XS),      % 2. Genera solo opciones válidas
    reverse(L1, L2),      % 3. INSTANCIA L2 primero
    append(L2,H,YS).      % 4. Usa L2 ya instanciada
```

**Por qué el orden malo no termina:**
- `append(L2,H,YS)` con L2 no instanciada genera infinitas opciones:
  - `L2=[], YS=...`
  - `L2=[_], YS=...`
  - `L2=[_,_], YS=...`
  - ... (infinito)
- Nunca alcanza `reverse(L1,L2)` que instanciaría L2
- Nunca alcanza `length(H,1)` que descartaría opciones incorrectas

**Regla de oro:** INSTANCIAR variables ANTES de usarlas en predicados generadores

**Orden correcto de metas:**
1. **Restringir** primero (con `length`, condiciones, guardas)
2. **Instanciar** variables mediante recursión u otros predicados
3. **Usar** las variables ya instanciadas en predicados generadores

### 2. Reglas Redundantes que Duplican Soluciones

**Problema:** Tener reglas que cubren casos ya manejados por otras reglas más generales causa soluciones duplicadas.

**Ejemplo: reverse/2 con regla redundante**

```prolog
% ❌ DUPLICA SOLUCIONES
reverse([], []).
reverse(X, X) :- length(X,1).    % ❌ Redundante con la regla general
reverse(XS, YS) :-
    length(H,1),
    append(H,L1,XS),
    reverse(L1, L2),
    append(L2,H,YS).

% Consulta:
?- reverse([1], X).
X = [1] ;      % Primera solución (regla 2)
X = [1].       % Segunda solución duplicada (regla 3)

% ✓ SIN DUPLICADOS
reverse([], []).
reverse(XS, YS) :-
    length(H,1),
    append(H,L1,XS),
    reverse(L1, L2),
    append(L2,H,YS).
```

**Regla de oro:** Eliminar reglas que son casos especiales ya cubiertos por reglas más generales, a menos que sean necesarias para eficiencia o terminación.

### 3. Checklist para Orden de Metas

Cuando escribas o analices predicados con múltiples metas:

1. ✓ **¿Las restricciones están primero?** (length, guardas, condiciones)
2. ✓ **¿Las variables se instancian antes de usarse?** (recursión antes de construcción)
3. ✓ **¿Los predicados generadores reciben variables instanciadas?**
4. ✓ **¿El caso base está completo?** ([], casos triviales)
5. ✓ **¿Hay reglas redundantes?** (verificar con consultas de prueba)
